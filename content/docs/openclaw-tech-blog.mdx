# OpenClaw：构建统一的多通道 AI 助手网关

## 引言：AI 助手集成的现实困境

在 AI 大模型快速发展的今天，企业和个人都希望将 AI 助手集成到日常工作流中。然而现实中的挑战却不少：

- **通道碎片化**：团队分散在 Telegram、Discord、WhatsApp、钉钉等多个平台
- **账号管理混乱**：工作号、个人号、测试号需要分别维护
- **隔离性差**：家庭成员共用一个 AI 助手，隐私和上下文容易混淆
- **扩展困难**：每增加一个通道就需要重新开发集成逻辑
- **安全风险**：直接暴露 AI 能力给所有用户，缺乏细粒度权限控制

这些问题让许多团队在 AI 助手落地时望而却步。OpenClaw 正是为解决这些痛点而生的统一网关框架。

---

## OpenClaw 是什么？

**OpenClaw** 是一个开源的多通道 AI 网关框架（MIT 许可证），通过单一 Gateway 进程统一管理多个消息通道、多个 AI agent、多个账号身份。它的核心理念是：

> **One Gateway, Many Agents, All Channels**

### 核心能力一览

- 🔌 **多通道支持**：原生支持 WhatsApp、Telegram、Discord、iMessage、Signal 等
- 🤖 **多 Agent 架构**：每个 agent 拥有独立的 workspace、身份、会话存储
- 🧩 **插件生态**：通过插件扩展 Mattermost、微信、钉钉等通道
- 📱 **移动节点**：iOS/Android 节点支持语音、相机、屏幕录制、定位等能力
- 🔐 **细粒度安全**：支持沙箱隔离、工具白名单、设备配对、OAuth 认证
- 🌐 **WebSocket 协议**：统一的实时通信协议，支持客户端、Web UI、移动节点

## 核心架构解析

### 1. Gateway：唯一的控制平面

OpenClaw 采用**中心化 Gateway** 设计，一个长期运行的进程负责：

- 维护所有消息通道连接（WhatsApp via Baileys、Telegram via grammY 等）
- 通过 WebSocket 协议对外提供统一 API（默认 `127.0.0.1:18789`）
- 管理 agent 路由、会话状态、设备配对

<Mermaid chart={`sequenceDiagram
    participant Client as 客户端/移动节点
    participant Gateway as OpenClaw Gateway
    participant WhatsApp
    participant Telegram
    participant Discord

    Client->>Gateway: WebSocket 连接 + 认证
    Gateway-->>Client: 握手成功 + 状态快照

    Gateway-->>WhatsApp: 维持长连接
    Gateway-->>Telegram: 维持长连接
    Gateway-->>Discord: 维持长连接

    Client->>Gateway: 发送消息请求
    Gateway->>Telegram: 推送消息
    Telegram-->>Gateway: 确认
    Gateway-->>Client: 事件流（streaming）`} />

**为什么选择中心化 Gateway？**

- **唯一会话管理**：一个 WhatsApp 账号只能由一个进程持有连接
- **统一协议**：客户端、Web UI、移动节点都使用相同的 WebSocket API
- **简化部署**：一个进程搞定所有通道，无需微服务编排

### 2. 多 Agent：独立的"大脑"

每个 **agent** 是一个完全隔离的 AI 实体，拥有：

| 组件 | 说明 | 示例路径 |
|------|------|----------|
| **Workspace** | 工作目录，包含 `SOUL.md`、`AGENTS.md`、`USER.md` 等人格文件 | `~/.openclaw/workspace-coding` |
| **agentDir** | 状态目录，存储认证配置、模型注册表 | `~/.openclaw/agents/coding/agent` |
| **Session Store** | 会话历史和路由状态 | `~/.openclaw/agents/coding/sessions` |

```json5
// ~/.openclaw/openclaw.json
{
  "agents": {
    "list": [
      {
        "id": "main",
        "workspace": "~/.openclaw/workspace-main",
        "model": "anthropic/claude-sonnet-4-5"
      },
      {
        "id": "coding",
        "workspace": "~/.openclaw/workspace-coding",
        "model": "anthropic/claude-opus-4-6",
        "tools": {
          "allow": ["exec", "read", "write", "browser"]
        }
      }
    ]
  }
}
```

**关键设计：**
- 每个 agent 的认证配置是独立的（`auth-profiles.json` 存储在各自的 `agentDir`）
- 技能（skills）可以是 agent 专属（`workspace/skills/`）或全局共享（`~/.openclaw/skills`）
- 工具权限可以按 agent 粒度配置（`allow`/`deny` 列表）

### 3. TypeBox 驱动的协议类型系统

OpenClaw 使用 **TypeBox** 定义 WebSocket 协议，自动生成：

- JSON Schema（用于运行时校验）
- Swift 模型（用于 iOS/macOS 客户端）

这保证了跨语言、跨平台的类型安全，避免了手写协议文档和模型的不一致问题。

## 多 Agent 路由机制

### 确定性路由规则

OpenClaw 的路由是**确定性**的，按照以下优先级匹配（最具体的规则优先）：

1. **peer 匹配**：精确的 DM/群组/频道 ID
2. **parentPeer 匹配**：线程继承（Discord/Slack 等）
3. **guildId + roles**：Discord 角色路由
4. **guildId**：Discord 服务器
5. **teamId**：Slack 工作空间
6. **accountId**：通道账号
7. **channel**：通道级匹配
8. **fallback**：默认 agent（`agents.list[].default` 或第一个）

### 示例 1：一个 WhatsApp 号，多个家庭成员

```json5
{
  "agents": {
    "list": [
      { "id": "dad", "workspace": "~/.openclaw/workspace-dad" },
      { "id": "mom", "workspace": "~/.openclaw/workspace-mom" }
    ]
  },
  "bindings": [
    {
      "agentId": "dad",
      "match": {
        "channel": "whatsapp",
        "peer": { "kind": "direct", "id": "+8613800138000" }
      }
    },
    {
      "agentId": "mom",
      "match": {
        "channel": "whatsapp",
        "peer": { "kind": "direct", "id": "+8613900139000" }
      }
    }
  ]
}
```

**效果**：同一个 WhatsApp 账号，根据发送者电话号码路由到不同的 agent，每个家庭成员有独立的上下文和记忆。

### 示例 2：多 Discord 机器人，分管不同服务器

```json5
{
  "agents": {
    "list": [
      { "id": "community", "workspace": "~/.openclaw/workspace-community" },
      { "id": "support", "workspace": "~/.openclaw/workspace-support" }
    ]
  },
  "bindings": [
    { "agentId": "community", "match": { "channel": "discord", "accountId": "bot-community" } },
    { "agentId": "support", "match": { "channel": "discord", "accountId": "bot-support" } }
  ],
  "channels": {
    "discord": {
      "accounts": {
        "bot-community": { "token": "DISCORD_TOKEN_1" },
        "bot-support": { "token": "DISCORD_TOKEN_2" }
      }
    }
  }
}
```

**效果**：两个 Discord 机器人账号，分别处理社区频道和客户支持频道，互不干扰。

## 实际应用场景

### 场景 1：工作/生活分离

**需求**：WhatsApp 处理日常琐事（用快速模型），Telegram 处理深度工作（用 Opus）。

```json5
{
  "agents": {
    "list": [
      {
        "id": "chat",
        "workspace": "~/.openclaw/workspace-chat",
        "model": "anthropic/claude-sonnet-4-5"
      },
      {
        "id": "opus",
        "workspace": "~/.openclaw/workspace-opus",
        "model": "anthropic/claude-opus-4-6"
      }
    ]
  },
  "bindings": [
    { "agentId": "chat", "match": { "channel": "whatsapp" } },
    { "agentId": "opus", "match": { "channel": "telegram" } }
  ]
}
```

### 场景 2：家庭群专用 agent，工具受限

**需求**：为家庭群绑定一个专属 agent，限制危险工具（如 `write`、`exec`），启用 mention 门控。

```json5
{
  "agents": {
    "list": [
      {
        "id": "family",
        "workspace": "~/.openclaw/workspace-family",
        "identity": { "name": "家庭小助手" },
        "groupChat": {
          "mentionPatterns": ["@小助手", "@家庭"]
        },
        "sandbox": {
          "mode": "all",  // 强制沙箱
          "scope": "agent"
        },
        "tools": {
          "allow": ["read", "web_search", "web_fetch"],
          "deny": ["exec", "write", "edit", "browser"]
        }
      }
    ]
  },
  "bindings": [
    {
      "agentId": "family",
      "match": {
        "channel": "whatsapp",
        "peer": { "kind": "group", "id": "120363012345678@g.us" }
      }
    }
  ]
}
```

**效果**：
- 只有 @小助手 或 @家庭 才会触发回复
- 工具能力受限（只能读文件、搜索网页）
- 所有操作在 Docker 沙箱中运行，无法访问宿主机敏感文件

### 场景 3：企业多部门 AI 助手

**需求**：研发部、市场部、客服部各自使用独立的 AI 助手，共享同一个 OpenClaw Gateway。

```json5
{
  "agents": {
    "list": [
      { "id": "dev", "workspace": "~/.openclaw/workspace-dev" },
      { "id": "marketing", "workspace": "~/.openclaw/workspace-marketing" },
      { "id": "support", "workspace": "~/.openclaw/workspace-support" }
    ]
  },
  "bindings": [
    { "agentId": "dev", "match": { "channel": "discord", "guildId": "111111111111111111" } },
    { "agentId": "marketing", "match": { "channel": "telegram", "accountId": "marketing-bot" } },
    { "agentId": "support", "match": { "channel": "whatsapp", "accountId": "support-line" } }
  ]
}
```

## 安全与沙箱

### 1. Docker 沙箱隔离

每个 agent 可以独立配置是否启用沙箱：

```json5
{
  "agents": {
    "list": [
      {
        "id": "untrusted",
        "sandbox": {
          "mode": "all",        // 始终启用沙箱
          "scope": "agent",     // 每个 agent 独立容器
          "docker": {
            "setupCommand": "apt-get update && apt-get install -y git"
          }
        }
      }
    ]
  }
}
```

**沙箱模式：**
- `off`：不启用沙箱（直接在宿主机运行）
- `all`：强制启用沙箱
- `scope: "agent"`：每个 agent 一个容器
- `scope: "shared"`：所有 agent 共享一个容器

### 2. 工具权限控制

通过 `tools.allow` 和 `tools.deny` 精细控制每个 agent 的能力：

```json5
{
  "agents": {
    "list": [
      {
        "id": "public-bot",
        "tools": {
          "allow": ["read", "web_search", "web_fetch"],
          "deny": ["exec", "write", "edit", "apply_patch", "browser", "nodes"]
        }
      }
    ]
  }
}
```

**常见工具分类：**
- **安全工具**：`read`, `web_search`, `web_fetch`, `session_status`
- **写入工具**：`write`, `edit`, `apply_patch`
- **执行工具**：`exec`, `browser`
- **高级工具**：`nodes`, `cron`, `gateway`

### 3. 设备配对与认证

所有 WebSocket 客户端（包括移动节点）必须完成**设备配对**：

1. 客户端首次连接时提供 `deviceId` 和签名
2. Gateway 验证签名，生成配对请求（可选自动批准本地连接）
3. 用户批准后，Gateway 颁发 `deviceToken`
4. 后续连接使用 `deviceToken` 快速认证

```bash
# 查看配对设备
openclaw pairing list

# 批准设备
openclaw pairing approve <requestId>

# 撤销设备
openclaw pairing revoke <deviceId>
```

## 扩展能力

### 1. 移动节点

OpenClaw 的 iOS/Android 节点通过 WebSocket 连接到 Gateway，提供丰富的设备能力：

- 📷 **相机**：拍照、录像（前置/后置）
- 📱 **屏幕录制**：录制手机屏幕
- 📍 **定位**：获取 GPS 位置
- 🔔 **通知管理**：列出、打开、回复、关闭通知
- 📞 **通讯录/日历/提醒事项**（iOS）
- 📧 **短信发送**（Android）

```bash
# 通过 nodes 工具调用移动节点能力
openclaw nodes camera_snap --node my-iphone --facing both
openclaw nodes location_get --node my-android
openclaw nodes screen_record --node my-iphone --duration 10s
```

### 2. Canvas 与 A2UI

Canvas 是 OpenClaw 内置的可视化能力，支持：

- Agent 动态生成 HTML/CSS/JS 界面
- A2UI（Agent-to-UI）协议，将 JSONL 指令转换为交互式 UI
- 移动节点上的 Canvas 渲染

```javascript
// agent 可以通过 canvas 工具呈现可视化内容
{
  "action": "present",
  "url": "https://example.com/dashboard",
  "width": 800,
  "height": 600
}
```

### 3. 插件生态

通过 `plugins.entries` 启用第三方插件：

```json5
{
  "plugins": {
    "entries": {
      "openclaw-weixin": { "enabled": true },
      "openclaw-mattermost": { "enabled": true }
    }
  }
}
```

插件遵循 OpenClaw Plugin SDK 规范，可以：
- 添加新的消息通道
- 注册自定义工具
- 监听 Gateway 事件

## 开发体验

### 快速启动

```bash
# 安装（需要 Node.js 18+）
npm install -g openclaw

# 初始化向导
openclaw onboard

# 启动 Gateway
openclaw gateway start

# 查看状态
openclaw status

# 登录通道（以 WhatsApp 为例）
openclaw channels login --channel whatsapp
```

### CLI 命令一览

```bash
# Agent 管理
openclaw agents list --bindings
openclaw agents add coding

# 通道管理
openclaw channels status --probe
openclaw channels login --channel telegram

# 配对管理
openclaw pairing list
openclaw pairing approve <requestId>

# 任务调度
openclaw cron list
openclaw cron add --name "Daily Report" --schedule "0 9 * * *"

# 配置管理
openclaw configure --section tools
openclaw gateway restart
```

### 配置文件结构

```plaintext

├── openclaw.json              # 主配置文件

│   ├── AGENTS.md

├── agents/
│   ├── main/
```

## 性能与可靠性

### 1. 会话压缩与剪枝

OpenClaw 内置会话压缩机制，避免上下文无限增长：

- **Safeguard 模式**：达到阈值时自动总结历史消息
- **手动总结**：通过 `/compact` 命令触发
- **自动剪枝**：长时间未活动的会话自动归档

### 2. 模型故障转移

支持多模型故障转移（failover）配置：

```json5
{
  "agents": {
    "defaults": {
      "model": {
        "primary": "anthropic/claude-sonnet-4-5",
        "fallback": ["openai/gpt-4", "google/gemini-pro"]
      }
    }
  }
}
```

### 3. 监控与日志

- **健康检查**：WebSocket `health` 请求返回 Gateway 状态
- **事件流**：所有操作生成事件（`agent`, `chat`, `presence`, `cron`）
- **日志级别**：支持 `debug`, `info`, `warn`, `error`

```bash
# 查看实时日志（systemd）
journalctl -u openclaw -f

# macOS launchd
tail -f ~/Library/Logs/OpenClaw/gateway.log
```

## 总结与展望

OpenClaw 通过**统一网关 + 多 Agent + 细粒度路由**的架构，解决了 AI 助手集成的核心痛点：

✅ **统一接入**：一个 Gateway 管理所有通道，告别重复开发  
✅ **隔离安全**：每个 agent 独立运行，支持沙箱和工具白名单  
✅ **灵活路由**：基于通道、账号、群组、用户的确定性路由  
✅ **易于扩展**：插件系统 + 技能市场，快速集成新能力  
✅ **生产就绪**：支持故障转移、会话压缩、设备配对等企业特性
