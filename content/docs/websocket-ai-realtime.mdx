# WebSocket 在 AI 时代找到了自己的位置

## 引言：沉寂多年，一夜翻红

如果你问我 WebSocket 诞生多少年了，我会告诉你：**2011 年就正式定稿了**。

十年间它一直是聊天室、在线游戏、实时通知的技术备选方案——重要但不起眼。直到大模型时代来临，Streaming AI 对话让"实时"变成了刚需，WebSocket 才真正站到舞台中央。

本文不讲 WebSocket 的协议细节（RFC 6455 随便一搜就有），我们聊点更实在的：**WebSocket 在 AI 应用中到底是怎么用的，以及为什么是它**。

---

## AI 对话为什么需要 WebSocket

### HTTP 的"请求-响应"困局

你跟 ChatGPT 聊天时，有没有注意到回复是一个字一个字蹦出来的？

这就是 **Streaming**——服务端分块返回数据，用户端实时渲染。但 HTTP 协议设计的是"一问一答"模式：

```
Client: 给我讲个笑话
Server: [处理中...]
Server: [处理完] 有一天...
```

如果用轮询（polling）模拟实时感，要么浪费资源（一直问"好了没"），要么延迟感人（等半天才拿到结果）。WebSocket 解决了这个问题：

```
Client: 给我讲个笑话
Server: [连接建立，保持]
Server: 发...
Server: 现...
Server: 一...
Server: 天...
Server: 笑...
```

### "正在输入"背后的技术

很多 AI 应用会显示"AI 正在思考..."或打字动画，这需要一个双向通道：

1. **Client → Server**：发送消息
2. **Server → Client**：发送状态更新（思考中、正在搜索、开始生成...）
3. **Server → Client**：分块发送 token

传统方案要用轮询或长轮询（long polling），资源浪费和延迟都难以接受。WebSocket 建立一次连接，这些问题迎刃而解。

## 实战场景一：流式对话（Streaming Chat）

这是最普遍的场景。我们用 Node.js + WebSocket 实现一个简化版：

```javascript
// 服务端：WebSocket 服务器转发 AI 流式响应
const WebSocket = require('ws');
const wss = new WebSocket.Server({ port: 8080 });

wss.on('connection', async (ws) => {
  ws.on('message', async (question) => {
    // 与 AI API 建立流式连接
    const aiStream = await openai.chat.completions.create({
      model: 'gpt-4',
      messages: [{ role: 'user', content: question.toString() }],
      stream: true,
    });

    // 将 AI 的流式响应透传给 WebSocket 客户端
    for await (const chunk of aiStream) {
      const token = chunk.choices[0]?.delta?.content || '';
      if (token) {
        ws.send(JSON.stringify({ type: 'token', data: token }));
      }
    }

    ws.send(JSON.stringify({ type: 'done' }));
  });
});
```

```javascript
// 客户端：实时渲染 AI 输出
const ws = new WebSocket('ws://localhost:8080');

ws.onopen = () => {
  ws.send('为什么天空是蓝色的？');
};

ws.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  
  if (msg.type === 'token') {
    // 流式追加到页面
    answerDiv.textContent += msg.data;
  } else if (msg.type === 'done') {
    console.log('回答完成');
  }
};
```

**效果**：用户看到文字逐字出现，像真人在打字一样。

## 实战场景二：AI Agent 的实时状态同步

在 Multi-Agent 系统里，多个 AI 需要协同工作。它们之间如何通信？

一个典型场景：**当 Agent A 处理完某任务，需要实时通知 Agent B 继续处理**。

```
[用户请求] 
     ↓
[Agent A: 理解意图]
     ↓（WebSocket 实时推送）
[Agent B: 执行操作]
     ↓（WebSocket 实时推送）
[Agent C: 生成回复]
     ↓
[返回用户]
```

```javascript
// Agent 事件总线 via WebSocket
class AgentBus extends WebSocket.Server {
  constructor() {
    super({ port: 8081 });
    this.agents = new Map();
  }

  // Agent 注册
  register(agentId, ws) {
    this.agents.set(agentId, ws);
  }

  // 向指定 Agent 发送消息
  sendTo(agentId, event) {
    const ws = this.agents.get(agentId);
    if (ws && ws.readyState === WebSocket.OPEN) {
      ws.send(JSON.stringify(event));
    }
  }

  // 广播给所有 Agent
  broadcast(event) {
    this.agents.forEach((ws) => {
      if (ws.readyState === WebSocket.OPEN) {
        ws.send(JSON.stringify(event));
      }
    });
  }
}

// 使用示例
bus.on('connection', (ws) => {
  ws.on('message', (data) => {
    const { agentId, event } = JSON.parse(data);
    bus.register(agentId, ws);
    
    if (event.type === 'task_complete') {
      // 通知下一个 Agent
      bus.sendTo('agent-b', {
        type: 'new_task',
        payload: event.result,
      });
    }
  });
});
```

## 实战场景三：多模态交互（语音/图像实时处理）

结合 WebSocket + WebRTC，AI 可以实现：

- **实时语音对话**：语音流通过 WebSocket 传输给 ASR 服务，识别结果再传回
- **屏幕共享 + AI 辅助**：用户屏幕实时流式传输，AI 分析并返回建议
- **图像标注**：上传图片后，AI 边看边说，哪里不对指哪里

```javascript
// 简化的实时语音转文字流程
const ws = new WebSocket('wss://api.example.com/asr');

// 假设我们用麦克风采集音频
navigator.mediaDevices.getUserMedia({ audio: true })
  .then((stream) => {
    const mediaRecorder = new MediaRecorder(stream);
    mediaRecorder.ondataavailable = (e) => {
      // 实时发送音频片段
      if (ws.readyState === WebSocket.OPEN) {
        ws.send(e.data);
      }
    };
    mediaRecorder.start(100); // 每 100ms 发送一次
  });

ws.onmessage = (event) => {
  const result = JSON.parse(event.data);
  transcriptDiv.textContent = result.text;
};
```

## 为什么不直接用 HTTP/2 或 gRPC？

有人会问：HTTP/2 支持多路复用，gRPC 支持 Streaming，干嘛非用 WebSocket？

| 特性 | WebSocket | HTTP/2 | gRPC |
|------|-----------|--------|------|
| 穿透防火墙/代理 | ✅ 天然支持 | ⚠️ 需要配置 | ❌ 通常不行 |
| 浏览器原生支持 | ✅ 普遍 | ✅ 普遍 | ❌ 需要插件 |
| 协议复杂度 | 低 | 高 | 中 |
| AI 生态集成 | ✅ 广泛 | ⚠️ 较少 | ⚠️ 较少 |
| 调试方便性 | ✅ ws:// URL 直接测试 | ❌ 需要工具 | ❌ 需要工具 |

AI 应用大量运行在浏览器环境，加上需要穿越各种企业网络，WebSocket 的兼容性和简单性是核心竞争力。

## 踩坑指南

### 1. 连接断开要重连

WebSocket 可能因为网络波动断开。要做好自动重连：

```javascript
function connect() {
  const ws = new WebSocket('ws://example.com');
  
  ws.onclose = () => {
    console.log('连接断开，5秒后重连...');
    setTimeout(connect, 5000);
  };
  
  ws.onerror = (err) => {
    console.error('WebSocket 错误:', err);
  };
}
```

### 2. 心跳保活

很多负载均衡器会在空闲连接超时后断开它。加心跳：

```javascript
// 服务端
const heartbeat = setInterval(() => {
  wss.clients.forEach((ws) => {
    if (ws.isAlive === false) return ws.terminate();
    ws.isAlive = false;
    ws.ping();
  });
}, 30000);
```

### 3. 消息压缩

AI 响应通常是文本，量大了很占带宽。可以开启 WebSocket 压缩：

```javascript
const wss = new WebSocket.Server({
  port: 8080,
  webSocketServerOptions: {
    perMessageDeflate: true,
  },
});
```

## 结语

WebSocket 不是什么新协议，但 AI 时代给了它新的使命。

它解决的不是"能不能"的问题，而是"体验好不好"的问题。当 AI 能实时思考、实时输出、实时交互，用户感受到的不只是一个工具，而更像是一个**活的东西在跟ta对话**。

这种感觉，很重要。
